## 1. Что это

`aiTelegaBot` — Telegram-бот, персональный **AI-дайджест**. Он собирает контент из
двух групп источников:

- **(а) Открытая веб-часть** — RSS-ленты, arXiv, Hacker News.
- **(б) Telegram-часть** — каналы/чаты, на которые подписан пользователь.

Собранное прогоняется через LLM (суммаризация + приоритизация) и отдаётся в
Telegram **сжатой сводкой в удобное время** (например, раз в день по расписанию).

Это первый проект портфеля с **настоящим сервисным рантаймом**: бот работает
постоянно (long-polling к Telegram + крон на дайджест), а не запускается как
локальный CLI/десктоп/IDE-плагин (в отличие от `gitl`/`shelf`/`orm-nplus1-radar`).
Отсюда — реальная (но дешёвая) инфраструктура: недорогой всегда-включённый VPS.

## 2. Зачем (и почему именно так)

Проект — **соло pet-проект с приоритетом обучения Go**. Это **не бизнес**:
экономическая целесообразность необязательна, «клиент» — сам автор. Ценность —
освоенные Go-навыки, работающий сервисный рантайм в портфолио и инструмент,
которым автор пользуется сам (dogfooding).

Ключевые принципы:

- **Golang-моно-стек, упор на написание кода**, а не «дёрнуть SDK». Максимум
  логики руками на stdlib + минимально необходимые библиотеки (см.
  `TECHNICAL_PLAN.md` §2).
- **BYOK** — пользователь подставляет свой LLM API-ключ; проект не хостит чужие
  ключи и не проксирует LLM.
- **Дешёвая, но реальная инфраструктура** — один недорогой VPS (~$5–10/мес),
  локальное состояние в SQLite. Без облачных БД/очередей/менеджедов.
- **Минимум общения от автора** — интроверт-friendly: бот обслуживает одного
  пользователя (себя), команды в чате + конфиг, без саппорт-контура.
- **Безопасность по умолчанию** — секреты (bot-токен, LLM-ключ, MTProto-сессия)
  никогда не в git; MTProto-сессия — самый чувствительный артефакт (см. §5).

## 3. Для кого

- **Прежде всего — сам автор**: единственный «пользователь» MVP. Бот шлёт дайджест
  в личный чат с ботом.
- Потенциально — другие интроверты-технари, которым надоело листать десятки лент
  и каналов, но публичный запуск/мультипользовательность **не цель MVP** (см.
  `POST_MVP_PLAN.md`).

## 4. Ключевые архитектурные решения (верхнеуровнево)

| Решение | Кратко | Подробности |
|---|---|---|
| Язык | Go 1.22+, упор на stdlib | §2 TECHNICAL_PLAN |
| Telegram (доставка/команды) | Bot API через `go-telegram/bot` | §2, §4 |
| Telegram (чтение приватных) | **MTProto через `gotd/td` — Go-моно-стек** (не TS-сайдкар) | §5 — решение и обоснование |
| Веб-источники | RSS/Atom через `mmcdole/gofeed`; arXiv/HN поверх него/через API | §3 |
| HTTP к LLM | ручной `net/http`-клиент, **без SDK** | §2 — тренировка retry/backoff |
| Планировщик | in-process крон на `time.Timer`/`robfig/cron` | §6 |
| Хранилище состояния | SQLite (файл на volume) через `modernc.org/sqlite` (без CGO) | §7 |
| Offline-режим | детерминированный экстрактивный суммаризатор без сети | dev/CI без ключа |
| Структура | `cmd/` + `internal/`, **без** `pkg/` | §3 — «не строй библиотеку, пока не просят» |
| Рантайм | docker-compose: сервис `bot` + volume состояния + опц. `ollama` | §8 |

## 5. Доступ к источникам Telegram и решение по MTProto

Тип доступа к источнику определяется **не хостингом сервера**, а тем, кем
пользователь является для источника:

| Тип источника | Метод | MTProto? | Фаза |
|---|---|---|---|
| Канал/группа, которой **управляет** пользователь | Bot API (бот — админ) | Нет | **1** |
| Группа, где пользователь рядовой участник (если настройки разрешают добавлять ботов) | Bot API | Нет | **1** |
| **Публичный** канал, пользователь просто подписан | Скрапинг `t.me/s/<channel>` (неофициально, может сломаться) | Нет | **1** (best-effort) |
| **Приватный** канал/группа, пользователь просто подписан (не админ) | User-сессия (MTProto) | **Да** | **2** |

**Решение по MTProto-стеку: остаёмся полностью на Go, используем `gotd/td`.**
Обоснование — в [`TECHNICAL_PLAN.md` §5](./TECHNICAL_PLAN.md). Коротко: `gotd/td` —
зрелая, активно поддерживаемая Go-библиотека MTProto; для личного проекта с низким
объёмом запросов её достаточно, а Go-моно-стек прямо служит цели «учить Go» (не
плодим второй рантайм на TypeScript+GramJS ради сайдкара).

**Риски MTProto (детально — `TECHNICAL_PLAN.md` §5, отложены в Фазу 2):**

- Это **реальный личный аккаунт**, не одноразовый бот → риск бана/ограничения бьёт
  по настоящему Telegram-профилю. Митигация: **отдельный вторичный аккаунт** под бота.
- **FloodWait** — жёсткие лимиты частоты (>~30 сообщений/сек уже триггерит) →
  обязателен exponential backoff + jitter, консервативная частота опроса.
- **Протокол меняется в июне 2026** (session-string v2; гранулярные
  `FLOOD_PREMIUM_WAIT_X`/`FLOOD_PEER_WAIT_X` вместо единого `FLOOD_WAIT_X`;
  депрекация части методов). Библиотеки обновляются за 1–3 недели — принимаем
  апдейт `gotd/td`, кастомную низкоуровневую реализацию не пишем.
- **Файл сессии = полный доступ к аккаунту** (не ограниченный скоуп, как у
  bot-токена) → вне git, в идеале зашифрован.
- Датацентр-IP на постоянно залогиненной сессии — доп.сигнал для анти-абьюз систем.

**Ключевое следствие для роадмапа:** MVP разделён так, чтобы ценность появилась
**до** любого касания MTProto. Фаза 1 полностью работает без user-сессии.

## 6. Этапы (от 0 до конца MVP)

Терминология репозитория — **«Этап N»** (не «недели»). Порядок важнее сроков.
Детализация действий/deliverable/критериев — в
[`TECHNICAL_PLAN.md` §9](./TECHNICAL_PLAN.md).

### Фаза 1 — ядро MVP (без MTProto вообще)

| Этап | Название | Однострочное описание |
|---|---|---|
| **0** | Bootstrap | Репозиторий, окружение, документация, каркас кода, `go build ./...` зелёный, Dockerfile + docker-compose. |
| **1** | Доставка в Telegram + команды | Bot API-клиент (`go-telegram/bot`), long-polling, `/start`/`/digest`/`/help`, отправка сообщения самому себе. |
| **2** | Веб-источники + модель контента | RSS/Atom (`gofeed`), arXiv, HN; нормализация в единый `Item`; дедуп по URL/ID в SQLite. |
| **3** | LLM-суммаризация + BYOK + offline | Ручной `net/http`-клиент (retry+backoff), сборка промпта, экстрактивный offline-суммаризатор, BYOK-конфиг. |
| **4** | Планировщик + сборка дайджеста | Крон (раз в день/настраиваемо), пайплайн collect→summarize→render→deliver, история отправленного в SQLite. |
| **5** | Telegram-источники через Bot API | Чтение каналов/групп, которыми пользователь управляет (бот-админ); best-effort скрапинг `t.me/s/<channel>` для публичных. **Конец Фазы 1.** |

### Фаза 2 — приватные Telegram-каналы через MTProto

| Этап | Название | Однострочное описание |
|---|---|---|
| **6** | MTProto-логин + безопасное хранение сессии | `gotd/td`: авторизация user-сессии (вторичный аккаунт), шифрованное хранение session вне git, разовый интерактивный логин. |
| **7** | Чтение приватных каналов + FloodWait-дисциплина | Опрос приватных каналов/чатов, exponential backoff + jitter, консервативная частота, единый `Item` из MTProto-сообщений. **Конец MVP.** |

## 8. Definition of Done (MVP, кратко)

- Бот стабильно работает как сервис на дешёвом VPS (docker-compose up), переживает
  рестарт (состояние в SQLite на volume).
- По расписанию приходит **один сжатый дайджест** из веб-источников и доступных
  Telegram-источников; дубликаты между запусками не повторяются.
- BYOK: подставляется свой LLM-ключ; без ключа — рабочий offline-режим.
- Bot API-путь (каналы/группы под управлением) и MTProto-путь (приватные каналы,
  вторичный аккаунт) оба протестированы на реальных источниках.
- MTProto-сессия хранится вне git, зашифрованной; FloodWait обрабатывается backoff'ом.

## 9. Бюджет

Разово: ≈ $0 (домен/сервисы не нужны). Ежемесячно: **дешёвый VPS ~$5–10/мес** —
единственная постоянная статья (первый проект портфеля с сервисным рантаймом).
Токены LLM платит пользователь своим ключом (BYOK); offline-режим и локальная
Ollama покрывают dev-цикл почти бесплатно. С запасом укладывается в лимиты
$500 разово / $50–70 в мес по всему портфелю.
