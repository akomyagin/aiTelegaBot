## 1. Nice-to-have (после MVP)

Пул задач, из которого берётся следующая по желанию/спросу. Не роадмап с датами.

### 1.1 Дополнительные источники

| Источник | Что даёт | Обучающая нагрузка Go |
|---|---|---|
| **Reddit** (сабреддиты через `.json`-эндпоинты или OAuth API) | Ниши, обсуждения | OAuth-флоу, rate-limit, пагинация |
| **YouTube** (новые видео каналов + субтитры/timedtext) | Видео-контент в текстовом дайджесте | RSS-ленты каналов + разбор субтитров |
| **GitHub Releases / Atom-ленты релизов** | Обновления интересующих проектов | Atom через gofeed, дедуп по тегу |
| **Mastodon / Bluesky** | Открытые соцсети без риска личного аккаунта | API-клиенты, курсорная пагинация |
| **Веб-страницы без ленты** (readability-извлечение) | Сайты без RSS | HTML→текст, эвристики извлечения статьи |

### 1.2 Дополнительные форматы дайджеста

| Формат | Что даёт | Обучающая нагрузка |
|---|---|---|
| **Кластеризация по темам** | Группировка похожих новостей, меньше шума | Эмбеддинги + простая кластеризация/дедуп по смыслу |
| **«Только важное» / приоритизация** | LLM ранжирует и отсекает малозначимое | Промпт-инжиниринг, пороги, персональные интересы в конфиге |
| **TL;DR + drill-down** | Короткая сводка + разворачивание по запросу | Stateful-команды бота, inline-кнопки, callback-роутинг |
| **Несколько расписаний/дайджестов** | Утренний веб-дайджест + вечерний Telegram | `robfig/cron`, несколько пайплайнов, конфиг профилей |
| **Экспорт** (Markdown-файл, отправка себе на email) | Архив дайджестов | Рендер в файл; email опционально через внешний SMTP пользователя |

### 1.3 Продуктовый полиш

| Фича | Что даёт | Обучающая нагрузка |
|---|---|---|
| **Веб-интерфейс настроек** (локальный, на том же VPS) | Управление источниками/расписанием без правки конфига | `net/http` + `html/template`, простая аутентификация; **строго single-user** |
| **Webhook вместо long-polling** | Экономнее на сервисе, «правильнее» продакшн | HTTPS-эндпоинт, secret-token, TLS/reverse-proxy |
| **Метрики/здоровье** (`/healthz`, счётчики) | Наблюдаемость сервисного рантайма | `net/http`, экспорт метрик (Prometheus-текст) |
| **Несколько получателей / шаринг** | Отправка дайджеста в общий канал/друзьям | Список chat_id; всё ещё без мультитенантности |

## 2. Отложенные риски (осознанно приняты на потом)

- **Хрупкость `t.me/s`-скрапинга** (Этап 5) — неофициальный публичный интерфейс,
  может сломаться в любой момент. В MVP деградирует мягко (лог + пропуск).
  Post-MVP: перенести публичные каналы на MTProto-путь, если скрапинг перестанет
  работать.
- **Смена MTProto-протокола** — принимаем апдейты `gotd/td`; если библиотека
  отстанет, вариант отступления — временный TS-сайдкар на GramJS только под
  MTProto-чтение (изолировано за интерфейсом `internal/mtproto`).
- **Стоимость LLM при росте числа источников** — post-MVP: кэш суммаризаций,
  батчинг, дешёвые модели/локальная Ollama, лимит числа элементов на дайджест.

## 3. Мультипользовательность (кандидат на v2 при спросе)

MVP — строго **на одного пользователя (себя)**. Превращение в сервис для многих
(регистрация, изоляция данных, чужие ключи/сессии) — большой отдельный проект и
**меняет профиль рисков и инфраструктуры**. Рассматривается только при явном
спросе и с пересмотром §4 (Never).

## 4. Excluded — Never (сознательно вне скоупа навсегда)

Эти пункты противоречат базовым принципам проекта («дешёвая single-user
инфраструктура», BYOK, минимум общения, безопасность аккаунта) и не рассматриваются:

- **Собственный LLM-прокси / хостинг чужих API-ключей** — конфликт с BYOK,
  расходы, приватность, ответственность за чужие ключи.
- **Платный SaaS с биллингом и поддержкой внутри бота** — превращает pet-проект в
  бизнес с саппорт-контуром (прямо против «минимум общения»).
- **Масс-скрапинг / чтение чужих приватных данных, спам-рассылки** — этически и по
  ToS Telegram недопустимо; бот читает только то, на что подписан **сам** владелец
  аккаунта, и шлёт дайджест **себе**.
- **Хранение MTProto-сессий нескольких людей на общем сервере** — недопустимая
  концентрация полного доступа к чужим аккаунтам.
- **Обход анти-абьюз/rate-limit Telegram** (множественные аккаунты-прокси, агрессивный
  поллинг) — сознательно консервативны, чтобы не жечь аккаунт.
