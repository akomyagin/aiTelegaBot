# CLAUDE.md

Инструкции для AI-сессий (Claude Code) в репозитории `aiTelegaBot`. Эти инструкции
имеют приоритет над поведением по умолчанию — следуй им точно.

## Что это за репозиторий

`aiTelegaBot` — Telegram-бот, персональный **AI-дайджест**: собирает контент из
RSS/arXiv/HN и Telegram-каналов, суммаризует через LLM и присылает сжатую сводку по
расписанию. Соло pet-проект с приоритетом **обучения Go**. **Не бизнес** —
экономическая целесообразность необязательна. Первый проект портфеля с настоящим
**сервисным рантаймом** (постоянно работает на дешёвом VPS).

Полное видение — [`docs/PLAN.md`](docs/PLAN.md); технический план и разбивка по
этапам — [`docs/TECHNICAL_PLAN.md`](docs/TECHNICAL_PLAN.md); за пределами MVP —
[`docs/POST_MVP_PLAN.md`](docs/POST_MVP_PLAN.md).

## Структура репозитория

| Путь | Содержимое |
|---|---|
| `docs/` | `PLAN.md`, `TECHNICAL_PLAN.md`, `POST_MVP_PLAN.md` |
| `.claude/skills/go-telegram-bot-dev/SKILL.md` | Конвенции именно этого проекта для написания кода |
| `cmd/bot/` | Тонкий `main.go` — context+signal, `config.Load`, `app.Run` |
| `internal/` | Всё ядро: `config/`, `feed/`, `llm/`, `telegram/`, `mtproto/` (Фаза 2), `digest/`, `scheduler/`, `storage/`, `app/` |
| `Dockerfile` | Multi-stage build статического Go-бинарника (CGO off) |
| `docker-compose.yml` | **Часть продукта:** сервис `bot` + volume состояния + опц. `ollama` (профиль `dev`) |

`pkg/` намеренно **нет** — всё ядро в `internal/`, чтобы никто не импортировал
незрелый API как библиотеку. Публичный API появится только при спросе.

## Модель-политика (какую модель на что вызывать)

Зафиксировано пользователем. Придерживайся строго.

- **Sonnet 5** — анализ документации и основной флоу чата (диалог по умолчанию, эта
  модель ведёт сессию); просмотр кода, проверка тестового покрытия, тестирование.
- **Opus 4.8** — написание и планирование, глубокие рассуждения, независимое ревью
  кода. Через **Agent-тул с `model: "opus"`**.
- **Fable 5** — написание кода по плану. Через **Agent-тул с `model: "fable"`**.

## Git / dev-workflow (обязательный процесс на каждый этап/задачу)

Главная ветка — **`master`** (не `main`).

1. **Новая ветка** от `master` на этап/задачу.
2. **Анализ** — предварительная подготовка (чтение `docs/`, `SKILL.md`, текущего
   кода) и подробный план реализации — основной чат (**Sonnet 5**); глубокая
   проработка/план — Agent **`model: "opus"`**.
3. **Написание кода** по плану — Agent **`model: "fable"`**.
4. **Просмотр** написанного кода — основной чат (**Sonnet 5**).
5. **Независимое ревью** — Agent **`model: "opus"`** (независимый от агента-кодера).
6. **Цикл исправлений:** если есть замечания — заново с шага 3 (код на **Fable 5**),
   **не более 3 циклов** шаги 3→5.
7. **Тестирование** — основной чат (**Sonnet 5**): `go build ./...`,
   `go vet ./...`, `gofmt`, `go test -race ./...`.
8. **Финальный просмотр + актуализация документации** в `docs/` — **Sonnet 5**.
9. **По завершении этапа:** commit, push ветки, создание **PR в `master`**.

Коммить/пуш только когда этап завершён (или пользователь явно попросил).

## Команды разработки

```bash
go build ./...                 # сборка (должна быть зелёной с Этапа 0)
go vet ./...                   # статические проверки
gofmt -l .                     # форматирование (должно быть пусто)
go test -race ./...            # тесты

# локальный запуск (long-polling); минимально нужен bot-токен
export TELEGRAM_BOT_TOKEN=... TELEGRAM_CHAT_ID=...
go run ./cmd/bot

docker compose up -d --build           # сервис bot + volume состояния
docker compose --profile dev up ollama # локальная Ollama для теста LLM без ключа
```

Перед коммитом кода: `go build ./...` и `go test ./...` должны проходить; желателен
`gofmt`/`go vet`.

## Ключевые архитектурные решения (не переоткрывай без причины)

- **Bot API** — библиотека `github.com/go-telegram/bot` (zero-dep, современная),
  спрятана за интерфейсом в `internal/telegram`. См. `TECHNICAL_PLAN.md` §4.
- **MTProto** (Фаза 2) — **Go-моно-стек на `github.com/gotd/td`**, НЕ TS-сайдкар на
  GramJS: зрелая библиотека + прямая польза для «учить Go» + один рантайм. См.
  `TECHNICAL_PLAN.md` §5.
- **LLM-доступ** — ручной `net/http`-клиент **без SDK** (тренировка retry/backoff);
  ключ **никогда** не логировать. Offline-суммаризатор — детерминированная эвристика.
- **Хранилище** — SQLite через `modernc.org/sqlite` (**без CGO**) + `database/sql`
  без ORM. Файл на volume.
- **Планировщик** — stdlib `time` в MVP; `robfig/cron` только когда нужен cron-синтаксис.
- Правило зависимостей: «интерфейс появляется на второй реализации, не на первой».

## Безопасность секретов (обязательно)

- `TELEGRAM_BOT_TOKEN`, `LLM_API_KEY` — только через env/`.env`. **Никогда не
  коммить.** `.env`, `*.db`, `*.session` уже в `.gitignore`.
- **MTProto session-файл (Фаза 2) — самый чувствительный артефакт: это ПОЛНЫЙ
  доступ к Telegram-аккаунту** (не ограниченный скоуп, как у bot-токена).
  - **Никогда не коммитить** (`*.session`/`*.session-journal` уже в `.gitignore` —
    но перепроверяй руками перед каждым коммитом Фазы 2).
  - Хранить **вне git**, на volume, с правами `0600`, в идеале **зашифрованным**
    (ключ из env, не в образе).
  - Использовать **отдельный вторичный Telegram-аккаунт**, а не основной личный —
    риск бана/ограничения от FloodWait/анти-абьюза реален. Детали и митигации —
    `TECHNICAL_PLAN.md` §5.1.
