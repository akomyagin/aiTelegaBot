# SKILL: go-telegram-bot-dev — конвенции проекта `aiTelegaBot`

Конкретные конвенции **именно этого проекта** для написания Go-кода. Это не общий
гайд «как писать Go», а специфика `aiTelegaBot`. Применяй при реализации любого
этапа. Опорные документы:
[`../../../docs/TECHNICAL_PLAN.md`](../../../docs/TECHNICAL_PLAN.md),
[`../../../docs/PLAN.md`](../../../docs/PLAN.md).

---

## 1. Структура: `cmd/` + `internal/`, почему нет `pkg/`

- `cmd/bot/main.go` — **тонкий**: строит `context` с `signal.NotifyContext`
  (SIGINT/SIGTERM → graceful shutdown сервиса), грузит конфиг, вызывает
  `app.Run(ctx, cfg)`. Никакой бизнес-логики в `main`.
- Всё ядро — в `internal/`, чтобы Go-компилятор **запрещал** внешний импорт: это
  приложение, а не библиотека. Пакеты: `config`, `feed`, `llm`, `telegram`,
  `mtproto` (Фаза 2), `digest`, `scheduler`, `storage`, `app`.
- **`pkg/` не заводить.** Публичный API — только при реальном спросе. Правило: «не
  строй библиотеку, пока её не просят».
- `internal/app` — единственное место «склейки»: собирает зависимости (store,
  telegram, llm, sources, scheduler) и запускает. Пакеты ядра друг о друге знают
  через интерфейсы, не напрямую.

## 2. Telegram Bot API: `go-telegram/bot`

Решение зафиксировано (`TECHNICAL_PLAN.md` §4): библиотека
`github.com/go-telegram/bot` (zero-dependency, context-aware, generics-хендлеры).

- Весь `go-telegram/bot` спрятан за интерфейсами в `internal/telegram`:
  `Deliverer` (отправка дайджеста) и `SourceReader` (чтение управляемых каналов).
  Если позже понадобится webhook/другая библиотека — меняется одна реализация.
- Long-polling запускается в своей горутине; **отмена — только через `context`**
  (тот же ctx, что и у всего сервиса). По ctx.Done() — корректно останавливаемся.
- Роутинг команд (`/start`, `/help`, `/digest`, `/sources`) реализуем **сами**
  (не прячемся за «магией» фреймворка) — это обучающая часть.
- **Bot-токен никогда не логировать.** Ошибки Telegram оборачивать, но без токена
  в тексте.
- Ограничение single-user: команды принимаем только от `TELEGRAM_CHAT_ID` из
  конфига (чужие апдейты игнорируем/логируем) — бот обслуживает только владельца.

## 3. MTProto (Фаза 2): `gotd/td`

Решение зафиксировано (`TECHNICAL_PLAN.md` §5): Go-моно-стек на
`github.com/gotd/td`, НЕ TS-сайдкар.

- Весь MTProto — за интерфейсом `mtproto.SourceReader` в `internal/mtproto`. Ядро
  (`digest`, `feed`) не знает про gotd — получает единый `feed.Item`.
- **FloodWait-дисциплина обязательна:** подключить floodwait+ratelimit миддлвары
  из `gotd/contrib`; поверх — собственный **exponential backoff + jitter**.
  Консервативная частота (раз в день, не поллинг). `FLOOD_*_WAIT_X` (в т.ч.
  гранулярные с июня 2026) обрабатываем ожиданием, **не** спамом retry.
- **Session storage:** файл вне git, на volume, права `0600`, **шифрование**
  (ключ из env, не в образе). Session = полный доступ к аккаунту — обращаться как
  с самым чувствительным секретом. Логин — разовый интерактивный (отдельный
  режим/флаг), **вторичный аккаунт**.
- При обновлении протокола (session-string v2 и т.п.) — **пинуем свежую `gotd/td`**,
  свой низкоуровневый клиент не пишем.

## 4. LLM-клиент: retry + backoff (ручной `net/http`, без SDK)

`internal/llm/client.go` — вручную на `net/http`, осознанно без SDK (тренировка).
Паттерн — как в `gitl`.

- Таймаут на запрос + общий `context`; отмена протянута насквозь.
- **Различать ошибки типизированно:** ретраебельные (429, 5xx, сетевые) vs
  фатальные (400, 401). `errors.Is`/типизированные ошибки, не строковое сравнение.
- **Retry с экспоненциальным backoff + jitter** (не фиксированная пауза), лимит
  `max_retries` из конфига.
- Секрет (`LLM_API_KEY`) **никогда** не логировать, даже в debug. Тест проверяет
  отсутствие ключа в stdout/stderr при ошибке.
- **BYOK:** ключ из env `LLM_API_KEY`; пустой → авто-offline (с предупреждением в
  лог, **не** падаем).

### Offline-provider паттерн

`internal/llm/offline.go` — детерминированная **экстрактивная** суммаризация без
сети.

- Реализует тот же интерфейс, что и сетевой клиент → подставляется прозрачно.
- Детерминизм обязателен: один вход → один выход (иначе тесты хрупкие). Например,
  отбор ведущих предложений/заголовков по простой эвристике.

## 5. Планировщик / крон

`internal/scheduler` — паттерн:

- MVP — stdlib `time`: считаем «следующий ежедневный слот» в заданной TZ
  (`time.LoadLocation`), ждём через `time.Timer`/`time.After` **с отменой по
  `context`** (никогда не блокирующий `time.Sleep` без ctx).
- **Идемпотентность слота при рестарте:** последний успешный прогон — в SQLite
  (`meta`); при старте проверяем, не пропущен/не выполнен ли сегодняшний слот.
  Не дублировать и не терять.
- Cron-синтаксис (`robfig/cron/v3`) — только когда реально понадобится несколько
  расписаний; раньше не тащить.

## 6. Хранилище: SQLite без CGO

`internal/storage` — `database/sql` + `modernc.org/sqlite` (чистый Go, **без CGO**
→ статический бинарник, простой Docker).

- Миграции — встроенные idempotent SQL (`CREATE TABLE IF NOT EXISTS`), версия в
  `PRAGMA user_version`, применяются при `Open`. Схема — `TECHNICAL_PLAN.md` §7.
- Без ORM — ручной SQL (обучающий эффект, контроль над схемой).
- Дедуп между запусками через `seen_items` (PK = dedup_key: URL/GUID или
  `tg:<channel>:<msgID>`). История — таблица `digests`.
- Все запросы с `context` (`QueryContext`/`ExecContext`), отмена сквозная.
- Один writer (single-user сервис) — но всё равно закрывать `rows`, проверять
  ошибки, использовать транзакции для многошаговых записей.

## 7. Тесты: table-driven + фейки + golden

- **Юнит:** парсеры `feed` (RSS/HN на фикстурах `testdata/`), дедуп-ключи, сборка
  промпта, расчёт слота планировщика, рендер дайджеста (**golden** в `testdata/`,
  сравнение байт-в-байт, обновление флагом `-update`).
- **Интеграционный ярус (обязателен, не только мок-HTTP):** детерминированные
  **фейки** (LLM-offline, фейковый `telegram.Deliverer`, фикстурный
  `feed.Source`) прогоняют **реальный** `digest.Pipeline.Run` поверх временной
  SQLite (`t.TempDir()`), проверяя дедуп/историю/идемпотентность слота.
- **`httptest.Server`** для retry LLM и HN-клиента (429→200, таймаут).
- Конкурентность (параллельный сбор источников) — прогонять с `-race`.

## 8. Общие правила

- Go **1.22+**: можно `slices`, `maps`, `log/slog`, `errors.Join`.
- Логи — `log/slog`; уровень из `LOG_LEVEL`/`--verbose`. **Секреты не логировать
  никогда** (bot-токен, LLM-ключ, MTProto-сессия).
- Секреты — только через env/`.env`; несекретные настройки — YAML. `.env`, `*.db`,
  `*.session` уже в `.gitignore`.
- Graceful shutdown: всё, что «крутится» (long-polling, scheduler), останавливается
  по `ctx.Done()`; закрывать store, дожидаться горутин.
- Интерфейс вводится **на второй реализации**, не на первой (анти-over-engineering).
- Каждая заглушка `internal/*` из Этапа 0 помечена комментарием «реализация — Этап
  N»; при реализации заменять содержимое, а не плодить параллельные файлы.
